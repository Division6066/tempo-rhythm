═══════════════════════════════
PART 1 — WHAT IS TEMPO?
═══════════════════════════════

TEMPO is a calm, minimalist, ADHD-friendly planning app that combines:
  - daily planning
  - tasks
  - simple notes
  - folders / areas / projects
  - tags
  - lightweight personal memory
  - AI-assisted prioritization
  - AI-assisted task chunking
  - AI-assisted planning from messy input

It should feel like a clean hybrid of NotePlan, Joplin, and a calm AI assistant.

It should NOT become:
  - a full Notion replacement
  - a giant PKM system
  - a full autonomous "agent OS"
  - a team collaboration platform
  - a giant second-brain app

The goal for THIS BUILD is Alpha: a genuinely useful internal prototype that
ships fast. Polish comes later. Shiny features come later.

═══════════════════════════════

PART 5 — ARCHITECTURE RULES
═══════════════════════════════

CRITICAL: Do NOT make the LLM or RAG the source of truth for live planner state.

Source-of-truth hierarchy (always apply this):
  1. User input wins
  2. Trusted external source wins over internal guesses
  3. Internal planner state wins over AI suggestions
  4. AI suggestions are advisory until the user explicitly accepts them

The architecture has four clearly separated layers:

LAYER A — Transactional planner state (the real database, always deterministic):
  tasks, notes metadata, projects, folders/areas, tags, daily plan,
  accepted AI outputs, recurrence state, reminders, external events if added later

LAYER B — Retrieval-backed memory (preferences and context, not live state):
  user preferences, routines, travel buffers, personal planning tendencies,
  recent relevant context, known constraints

LAYER C — Template / primitive layer (a small fixed library of known blocks):
  heading, checklist, task section, top-3 section, focus block,
  study session block, notes section, reflection block, habit section

LAYER D — AI helper layer (advisory only, never writes directly to planner state):
  The AI parses messy input, extracts tasks, suggests priorities,
  chunks large work, rewrites into calmer plans, generates from templates,
  retrieves relevant memory when appropriate.

  The AI NEVER:
  - invents commitments
  - silently overwrites user intent
  - hallucinates schedule facts
  - mutates planner state without user visibility

PART 11 — AI BEHAVIOR RULES
═══════════════════════════════

For alpha, AI uses mock responses only. Leave clean TODO comments everywhere
a real OpenRouter call should go. Use the openrouter-chat template as the
reference pattern for wiring in real AI later.

AI features to mock in alpha:
  - Plan generation from task list + preferences
  - Task extraction from messy text (brain dump input)
  - Prioritization suggestions
  - Task chunking (large task → subtasks)
  - ADHD-friendly rewrite (simplify language, reduce overwhelm)

Mock behavior pattern:
  User submits input → 1800ms delay → breathing-light indicator →
  show response from aiResponsePool cycling → user can accept/edit/reject

The accept/edit/reject pattern is CRITICAL. AI output always goes into a
staging area first. It only reaches the planner state when the user taps Accept.
